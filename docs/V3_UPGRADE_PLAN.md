### 1.2 V2 已知局限

1. **单食堂单楼层**：只能模拟明湖食堂一层，不支持多楼层或多食堂对比
2. **Agent 行为简单**：固定路径模式，无社交行为、无组队就餐、无中途改变决策
3. **无历史数据回放**：仿真结果无法保存和回放
4. **无参数优化**：缺少自动寻找最优窗口配置的功能
5. **行为树未集成**：`behavior-system/` 框架已完成但未接入引擎
6. **性能瓶颈**：Agent > 200 时 VoxelAgent 逐个渲染开始吃力

## 二、V3 升级目标

### 2.1 核心定位

**从"仿真展示工具"升级为"调度决策支持系统"**

V2 侧重视觉呈现，V3 侧重决策能力——用户能通过仿真实验回答"怎么配置窗口最优？"、"高峰期需要多少座位？"等实际运营问题。

### 2.2 目标用户场景

1. **食堂管理者**：评估不同窗口方案的效果，制定高峰期调度策略
2. **课程教学**：演示排队论、ABM 方法论，支持参数探索实验
3. **作品集展示**：在线可交互，一键 Demo，视觉冲击力强

## 三、功能升级计划

### 3.1 P0：行为树引擎集成

**目标：** 将已实现的 `behavior-system/` 接入仿真引擎，替换当前 FSM 中的部分逻辑。

**设计方案：**

```
当前：FSM executor + 手动窗口队列管理（混合模式）
目标：BT 作为顶层决策 + FSM 管理底层状态转移

Agent.tick(dt):
  1. BT.evaluate()  → 决策层（选窗口/是否放弃/是否换窗口）
  2. FSM.execute()  → 执行层（状态转移 + 副作用）
```

**新增行为：**
- 排队中途换窗口（发现旁边窗口更短时动态切换）
- 组队就餐（2-3人结伴，选择同一窗口或相邻座位）
- 犹豫行为（在入口观察几秒再决定）

**工作量估算：** 3-4天

**风险：** BT 与现有 FSM 的交互边界需要仔细设计，避免状态冲突。需要增加 20+ 个测试用例。

---

### 3.2 P0：A/B 对比模式

**目标：** 支持左右分屏同时运行两种配置方案，直观对比效果。

**设计方案：**

```
┌──────────────────────┬──────────────────────┐
│   方案 A：6 窗口      │   方案 B：8 窗口      │
│   相同到达曲线         │   相同到达曲线         │
│   [3D 场景]           │   [3D 场景]           │
│   avgWait: 23.4s      │   avgWait: 16.1s      │
│   abandon: 8.2%       │   abandon: 3.5%       │
└──────────────────────┴──────────────────────┘
                    对比结论面板
          "方案 B 等待时间降低 31%，放弃率降低 57%"
```

**技术方案：**
- 两个独立的 `SimulationEngine` 实例，共享相同的随机种子
- 两个 `<Canvas>` 或单个 Canvas 内 viewport 分割
- 对比指标面板居中显示差异百分比

**工作量估算：** 2-3天

---

### 3.3 P1：参数优化器

**目标：** 自动搜索最优配置参数（窗口数量、服务时间分配），最小化目标函数（如 P95 等待时间）。

**设计方案：**

```typescript
interface OptimizationConfig {
  objective: 'avgWaitTime' | 'waitTimeP95' | 'abandonRate' | 'throughput';
  constraints: {
    maxWindowCount: number;   // 最多能开几个窗口
    maxBudget: number;        // 总成本预算
    minThroughput: number;    // 最低吞吐要求
  };
  method: 'grid-search' | 'bayesian' | 'genetic';
  maxIterations: number;
}
```

**搜索策略：**
1. **网格搜索**：适合 2-3 个参数的小规模探索
2. **贝叶斯优化**：适合参数空间较大时的高效搜索
3. **遗传算法**：适合多目标优化（Pareto 前沿）

**结果展示：** 优化过程动画（显示每次迭代的参数和目标值变化），最终输出最优方案。

**工作量估算：** 5-7天

---

### 3.4 P1：仿真回放与数据导出

**目标：** 保存仿真运行记录，支持回放和数据分析。

**设计方案：**

```typescript
interface SimulationRecord {
  config: SimulationConfig;
  frames: FrameSnapshot[];  // 每0.5s一个快照
  finalStats: SimStats;
  metadata: { timestamp, duration, seed };
}

interface FrameSnapshot {
  time: number;
  agents: { id, state, position }[];
  stats: SimStats;
}
```

**功能：**
- 录制：仿真运行时自动录制帧快照
- 回放：时间轴拖动，快进/慢放
- 导出：CSV/JSON 格式导出统计数据
- 对比：加载两次录制结果进行对比分析

**工作量估算：** 3-4天

---

### 3.5 P1：多楼层/多食堂支持

**目标：** 支持配置多个食堂或多楼层场景，Agent 可在不同区域间流动。

**设计方案：**

```typescript
interface CampusDiningConfig {
  canteens: {
    id: string;
    name: string;
    scenario: SimulationScenario;
    position: Vec2;  // 在校园地图中的位置
  }[];
  flow: {
    // 不同食堂间的客流分配比例
    [fromCanteen: string]: { [toCanteen: string]: number };
  };
}
```

**可视化：** 校园俯视图 → 点击某个食堂进入详细仿真视图

**工作量估算：** 5-7天

---

### 3.6 P2：性能优化 — InstancedMesh Agent 渲染

**目标：** Agent 数量 > 200 时切换到 InstancedMesh 批量渲染，支持 500+ Agent 流畅运行。

**设计方案：**

当前 VoxelAgent 逐个渲染（每个 Agent 4 个 mesh），200 Agent = 800 draw calls。

优化方案：
1. 身体/头/腿分别用 3 个 InstancedMesh（最多 500 实例）
2. 颜色通过 instanceColor attribute 逐实例设置
3. 行走动画通过修改 instanceMatrix 实现
4. 未使用的实例 scale=0 隐藏

**预期效果：** 从 ~800 draw calls 降到 ~3 draw calls，支持 500 Agent @ 60fps。

**工作量估算：** 2-3天

---

### 3.7 P2：移动端适配

**目标：** 响应式布局 + 触摸交互，在手机/平板上可演示。

**改造点：**
- 控制面板改为底部抽屉式
- 统计面板改为可展开/折叠
- 相机操作支持双指缩放/单指旋转
- Demo Mode 添加全屏按钮
- 性能降级：移动端减少热力图粒子、降低阴影精度

**工作量估算：** 3-4天

---

### 3.8 P2：实时协作（WebSocket）

**目标：** 多人同时查看同一仿真场景，教师端可控制参数变更。

**架构：**

```
教师端 (Controller)     学生端 (Viewer) × N
     │                        │
     └───── WebSocket ────────┘
              │
        共享仿真状态
```

**技术方案：**
- 服务端运行仿真引擎，定期广播帧状态
- 客户端只负责渲染，不运行仿真逻辑
- 教师端可实时调参（修改到达率、增减窗口）

**工作量估算：** 7-10天

## 七、风险评估

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| BT-FSM 集成产生状态冲突 | 中 | 高 | 先小范围集成（只加换窗口行为），充分测试后扩展 |
| 优化器搜索空间爆炸 | 低 | 中 | 限制参数维度，优先实现网格搜索 |
| InstancedMesh 动画复杂度高 | 中 | 低 | 可降级为静态颜色方块（去掉行走动画） |
| 多食堂 Agent 跨区域路径规划 | 中 | 中 | 简化为概率分配，不做真实路径规划 |

---

## 八、总结

V3 的核心升级方向是从"看"到"用"——让仿真不仅能展示，还能辅助决策。行为树集成让 Agent 行为更真实，A/B 对比和参数优化让系统具备实际分析价值，回放和导出让结果可追溯可分享。

优先实现 P0（行为树 + A/B 对比），这两个功能投入产出比最高，既提升学术深度，又增强演示效果。
